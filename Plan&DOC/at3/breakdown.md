# COM668 Assessment Task 3 – Project Review Report

This document outlines the structure and key requirements for the AT3 Project Review Report, drawing from the "Assessment Overview - Semester 2" and "AT3 Report Template(1)" documents, as well as the additional detailed breakdown provided.

## 1. Report Purpose and High-Level View

[cite_start]Assessment Task 3 (AT3) results in a technical report that documents the software development challenge addressed during the module. [cite_start]It provides a review of the work carried out since that covered by the AT2 report and evidence of your learning during the module. [cite_start]This report is a required deliverable and must be submitted to pass the module. [cite_start]Assessment is agreed by your Mentor and your allocated marker. [cite_start]The AT3 rubric has detailed examples of what is expected at each level of attainment.

The report should:
* [cite_start]Update the agreed concept, aim and objectives, justifying any changes.
* [cite_start]Present the detailed plan of work undertaken, discussing metrics and justifying decisions.
* [cite_start]Detail the technical design of the software that you then went on to develop.
* [cite_start]Detail the verification, validation and testing of the resultant software.
* [cite_start]Evaluate your process and the results it produced, in context.
* [cite_start]Draw conclusions from your project.
* [cite_start]Present a code manifest table as an appendix of the report.

[cite_start]The maximum length for the report is 9,000 words.

## 2. Word Count Policy

[cite_start]The maximum word count for the report is 9,000 words. [cite_start]You are required to self-declare the word count in this report. [cite_start]A falsified word count may be considered as academic misconduct. [cite_start]This is in keeping with Ulster University policy.

[cite_start]Ulster University has updated its policy on maximum word counts.
* [cite_start]Students must self-declare the word count for a submission.
* [cite_start]Work under the word limit is less likely to fully meet assessment criteria.
* [cite_start]There is flexibility up to 10% over the stated maximum word limit.
* [cite_start]Work will only be considered and marked up to that point.
* [cite_start]A falsified word count may be considered as academic misconduct.

During COM668, the maximum word counts **exclude**:
* [cite_start]Title pages.
* [cite_start]Contents.
* [cite_start]Reference lists.
* [cite_start]Code Manifest.

[cite_start]Everything else counts towards the word count, including any appendices. [cite_start]Further guidance on word counts is available in the Module Handbook.

### 3.3. Introduction
[cite_start](Suggested word count for Introduction: 500) 
* [cite_start]An updated description of the agreed concept.
* [cite_start]Restate the aim and objective of the project.
* [cite_start]Justify any changes to the aim since assessment task 2.
* [cite_start]Briefly describe the achieved outcome, including whether the aim was achieved.

### 3.4. Plan
(Suggested word count for Plan: 1000 words only)
* [cite_start]Provide the plans used to manage the project.
* [cite_start]Discuss metrics used to monitor and review progress against the plan.
* [cite_start]Justify the planning decisions made, in context of the methodology.

### 3.5. Technical Solution
(Suggested word count for Technical Solution: 3000 words only)
* [cite_start]Select an appropriate software architecture, for example C4.
* [cite_start]Detail the system context, container, component and code design.
* [cite_start]Detail the data design process and outcomes.
* [cite_start]Detail the user experience design considerations, options and selected designs.
* [cite_start]Consider Data security and Accessibility factors.

### 3.6. Testing, Verification and Validation
(Suggested word count for Testing, Verification and Validation: 1800 words only)
* [cite_start]Test the software product against the requirements, ensuring good coverage, e.g., multiple tests per requirement.
* [cite_start]Verify the work products.
* [cite_start]Validate the software product.

### 3.7. Evaluation
(Suggested word count for Evaluation: 1800 words only)
* [cite_start]Evaluate the overall achievement of the project (successes and failures).
* [cite_start]Evaluate the effectiveness of the plan and approach to management of the project.
* Reflect on the adequacy of the software methodology employed.
* [cite_start]Evaluate the value of the tools used during the technical development of the work.
* [cite_start]Evaluate any innovative aspects of the project.
* [cite_start]Evaluate the commercial, academic and social contexts as appropriate, perhaps including: data security considerations, accessibility factors, legal limitations, social factors, ethical considerations, environmental impact.

### 3.8. Conclusion
(Suggested word count for Conclusion: 400 words only)
* [cite_start]Summarise the report and any significant conclusions.
* [cite_start]Reflect on your learning journey.

### 3.9. References
(Section not included in word count)
* [cite_start]PROPER CITATION AND REFERENCING IS CRUCIAL.
* [cite_start]FAILURE TO PROPERLY ACKNOWLEDGE THE WORK OF OTHERS OR THE USE OF ARTIFICIAL INTELLIGENCE (AI) IS ACADEMIC MISCONDUCT, WHICH CAN RESULT IN FAILURE.
* [cite_start]If in doubt, consult your Mentor.
* [cite_start]Read the relevant University policies (links in Module Handbook).
* [cite_start]Cite work that is not yours at the point of inclusion in the report.
* [cite_start]List the full details of all items cited within the report.
* [cite_start]Presented references in the Harvard style (see Library guide).
* [cite_start]Take care to cite all work used or referred to, no exceptions.
* [cite_start]Acknowledge Artificial Intelligence (AI) generated content.
* List of full references for all items used or cited within the report, presented in in the Harvard style.

### 3.10. Code Manifest Appendix
(Section not included in word count)
* [cite_start]A table that identifies the source code files you created or contributed to.
* It should show:
    * [cite_start]File name.
    * [cite_start]File path (location) within the source code zip file.
    * Nature of your contribution to the file, i.e., created, modified, based on tutorial, etc.
    * [cite_start]Purpose of file within the software product.
* [cite_start]Note, do not include listings of the actual source code, that is provided in the zip file.

### 3.11. Additional Appendices
(Suggested word count for Additional Appendices, if needed: 500 words only)
* [cite_start]Include if required.
* [cite_start]For example, further designs or lists of further test cases.
* [cite_start]However, include and discuss a few samples in the report body.

## 4. Report Format

The following is a guide to what works well. Staying close to it is highly recommended. [cite_start]The report should be presented in a professional manner.
[cite_start]Format details are in the handbook and include:
* A4 paper size.
* Single Column layout.
* 2cm margins all around.
* 1.25 line spacing.
* Arial for titles.
* Times New Roman for body text.
* Courier for code.
* 11 or 12 point font for body text, headings a little larger.
* Page numbering at the bottom of each page.

* [cite_start]Submit in PDF format (not Word or otherwise).
* [cite_start]Filename format: `surname_firstname_B00123456.pdf`.

## 5. Source Code Submission

* [cite_start]Submit all source code noted in the code manifest as a separate zip file.
* [cite_start]Keep the files in their original format.
* [cite_start]Ensure ownership of code is clearly indicated in comments, especially for any third-party code.
* [cite_start]Filename format: `surname_firstname_B00123456.zip`.
